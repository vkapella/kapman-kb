# SIGNAL

## Principle

SIGNAL owns the named-trigger vocabulary that converts regime reads into runtime decisions. Where WYCKOFF, DEALER, and VOLATILITY each surface a regime as a tier label, a flag, or a band, SIGNAL composes those outputs into the specific firing conditions that govern eligibility, structure, and exit — and into the behavioral consequences that firing produces. The governing judgment is that triggers are vocabulary, not algorithms: a trigger names a regime composite, defines what must be true for it to fire, defines what fires means, and names the downstream file that enforces the consequence. Triggers do not re-derive regime; they consume it. The runtime contract for every named trigger has four parts — what regime inputs it consumes, what condition makes it fire, what behavioral consequence firing produces, and which downstream file owns the enforcement venue — and the trigger is documented in this file in those terms. Three behavioral axes structure the trigger set. Entry triggers gate whether a candidate makes the eligible set at all and are enforced at PASS1 and PASS2; they include the Wyckoff veto (an entry refused because the confirmed Wyckoff regime does not authorize the candidate's direction) and the dealer-timing veto (an entry refused because the regime is too unstable to support new exposure at this moment). Structure triggers gate which structure is permitted for an otherwise-eligible candidate; the canonical structure trigger is the spread-mandate, formerly known as `iv_forbids_long_premium` — when forward IV reads materially richer than realized vol via the IV/HV elevated band owned by VOLATILITY, naked long-premium is refused and a vertical spread becomes mandatory for new directional entries. Exit triggers govern when an open position should be reviewed or closed; they include hard-trigger broker-actionable alerts (a Stop alert tied to an underlying price level where the thesis breaks down, a Profit target alert tied to an underlying price level where the thesis matures) and an advisory regime-exit flag that surfaces when the regime that authorized the original entry has decayed materially but does not force a close. The three axes are surfaced separately in the report because they answer different operator questions: entry triggers answer "can this trade happen at all," structure triggers answer "in what form," exit triggers answer "what's the operator's plan if it stops working." Trigger inputs come from two sources — MCP-delivered regime outputs (dealer regime per DEALER, volatility regime per VOLATILITY) and operator-confirmed regime judgments (Wyckoff regime, phase, and events per WYCKOFF, where the runtime proposes a regime/phase-and-event reading from MCP-delivered price/volume metrics and the operator confirms or corrects before the reading is consumed) — and never derived inside SIGNAL; when an input regime is degraded (LIMITED, INVALID, or stale per the upstream principle file, or unconfirmed by the operator in the case of Wyckoff), the dependent trigger is degraded to the conservative read: entry triggers default to refusal, structure triggers default to spread-mandated, exit triggers surface a *Pending* annotation rather than firing or staying silent. SIGNAL preserves the anti-hallucination floor inherited from `KAPMAN_GUARDRAILS_v4.0.md` — when a trigger would require an exact strike, expiration, or numeric target that has not been validated against an MCP options-chain call, the trigger output uses zones or qualitative labels, never invented specifics. Override authority over any mandate trigger remains the operator's per GUARDRAILS; SIGNAL enforces the trigger contract, GUARDRAILS enforces the override discipline, and the two work together so that the trigger doesn't silently relax and the operator's override doesn't silently change behavior.

## Operational heuristics

**Heuristic 0 — Near-event-risk veto (fires before all other trigger evaluation)**

The earnings date this heuristic consumes is fetched live from `Finnhub MCP Server:get_earnings_calendar` for the candidate ticker, with a query window from the screening date through the screening date plus `EARNINGS_CAUTION_DAYS` plus a 7-day buffer, so a next-earnings date sitting just beyond the caution window surfaces as context rather than silence. Days are counted by calendar date alone; the reported session hour (`bmo`/`amc`/`dmh`) rides in the rationale as context and never shifts the day count. An operator-declared earnings date stated in the current session takes precedence for that session and is labeled *declared* rather than *validated* in the data-quality surface. Model-internal knowledge is never an earnings-date source — a date that cannot be produced by the tool or the operator is unavailable, per the anti-hallucination floor inherited from `KAPMAN_GUARDRAILS_v4.0.md`.

When a confirmed earnings date falls within `EARNINGS_BLOCK_DAYS` (per SYSTEM_PARAMS, currently 7 calendar days) of the screening date, the near-event-risk veto fires unconditionally. Output = WAIT. Type = NONE. Rationale = "Near event risk — earnings [date] ([session hour]) within [N]d." All other regime reads (Wyckoff, dealer, volatility, macro) are skipped for this candidate.

When a confirmed earnings date falls within `EARNINGS_CAUTION_DAYS` (per SYSTEM_PARAMS, currently 21 calendar days) but outside `EARNINGS_BLOCK_DAYS`, the near-event-risk caution fires. Output = WAIT with operator-approval gate. Rationale = "Earnings [date] ([session hour]) — [N]d out. Operator approval required to advance to Eligible." The candidate does not enter the Eligible set until the operator explicitly redirects it in the current session.

When the tool query returns no earnings date inside the window, the veto does not fire and the absence is a validated read, not a blind spot. Surface in data-quality output: "No earnings within [EARNINGS_CAUTION_DAYS]d — validated via Finnhub [query timestamp]."

When the tool is unavailable or errors and no operator-declared date exists, the screen cannot be evaluated — and a veto that cannot be evaluated never silently passes. The candidate receives WAIT with the same operator-approval gate as the caution window. Rationale = "Earnings screen degraded — Finnhub unavailable. Operator approval required to advance to Eligible." The prior v4.0 behavior (no date → veto does not fire, "Earnings date not provided") is retired; that silent pass is exactly the gap the named source closes.

**Wyckoff veto — entry refused when the confirmed Wyckoff regime does not authorize the candidate's direction.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Operator-confirmed Wyckoff **regime** and **phase (A–E)** for the candidate ticker (per WYCKOFF's propose-confirm protocol); the candidate's **resolved direction** (BULLISH/BEARISH, per heuristic 11 or operator declaration); operator-confirmed **phase-C confirmation** status — `spring`/`shakeout` (bullish phase-C, accumulation-family) or `utad` (bearish phase-C, distribution-family) — for the regimes that gate on it |
| Firing condition | The confirmed regime does not authorize the candidate's resolved direction. **Bullish** candidate fires when regime ∈ {`distribution`, `redistribution`, `markdown`} (structural), OR regime ∈ {`accumulation`, `reaccumulation`} without a confirmed bullish phase-C (`spring`/`shakeout`) (conditional). **Bearish** candidate fires when regime ∈ {`accumulation`, `reaccumulation`, `markup`} (structural), OR regime ∈ {`distribution`, `redistribution`} without a confirmed bearish phase-C (`utad`) (conditional). **Either** direction fires when regime = `ranging_undefined` (confirmed stand-aside) OR no operator confirmation exists (`UNKNOWN`). The veto stays **silent** when the regime authorizes the direction: `markup` (bullish) / `markdown` (bearish) confirmed trends, or the matching range regime with its phase-C confirmed |
| Behavioral consequence | New long-premium entry **in the candidate's direction** refused — dropped from the eligible set for that direction (long calls + call debit spreads for a bullish candidate; long puts + put debit spreads for a bearish candidate). The redirect surfaces what the prevailing regime *does* authorize: the directionally-aligned long-premium for the regime (long puts in `markdown`/`redistribution`; long calls in `markup`/`reaccumulation`), plus CSPs, hedges, and LEAPs per GUARDRAILS' eligible-set redirect |
| Enforcement venue | `PASS1_SCREENING_v4.0.md` enforces the refusal at eligible-set determination; `REPORT_FORMAT_v4.0.md` renders the redirect language |

The Wyckoff veto is the strongest entry-side gate in the system because Wyckoff regime identification is the most regime-defining input the runtime consumes — it's what determines whether a long-premium structure has structural tailwind, structural headwind, or no clear directional context. The veto is **direction-aware and symmetric**: it asks whether the confirmed regime authorizes the candidate's resolved direction, so the bullish and bearish cases mirror each other (a `markdown` regime that refuses a long call is exactly the regime that authorizes a long put, and vice versa). It fires conservatively: an unconfirmed Wyckoff reading (`UNKNOWN`) refuses long-premium in **either** direction, because acting on an unconfirmed read against the runtime's strongest gate is the failure mode this trigger exists to prevent. Operator override is available per GUARDRAILS' override discipline, but the override must explicitly name the Wyckoff veto being relaxed and must accompany an explicit regime/phase declaration that supersedes the unconfirmed status. The refusal has three distinct characters, and the report language distinguishes them: (1) **structural** refusal — the regime favors the opposite direction (a bullish candidate in `distribution`/`redistribution`/`markdown`, or a bearish candidate in `accumulation`/`reaccumulation`/`markup`) surfaces as *"Wyckoff regime refuses [direction] long-premium — eligible set redirected"*; (2) **conditional** refusal — the matching range regime without its phase-C is a *"wait for confirmation"* read, not *"this candidate is structurally wrong,"* and surfaces as *"Wyckoff veto — [regime] pre-phase-C; awaiting [`spring`/`utad`] confirmation"*; (3) **stand-aside** — a confirmed `ranging_undefined` surfaces as *"Wyckoff regime ranging/undefined — stand aside."* The conditional branch is preserved separately because its structural meaning differs and because phase-C confirmation (`spring`/`shakeout` on the bullish side, `utad` on the bearish side), not a regime change, is what releases it. On the viewer-ingest path, phase-C confirmation is read deterministically from the row's own event/setup evidence, and **completed-phase evidence counts**: the conditional branch releases on the phase-C event itself (`spring`/`shakeout` bullish; `utad`/`ut` bearish) *or* on later-phase evidence that presupposes a completed phase C — bullish `setup_tags` any of `phase_c_spring_long` / `phase_d_lps_long` / `sos_breakout_long`, or `last_event` `sos`/`jac`; bearish `setup_tags` any of `phase_c_utad_short` / `lpsy_short` / `sow_breakdown_short`, or `last_event` `sow`. A phase-D LPS or an SOS breakout cannot exist without the phase-C test that preceded it, so demanding the phase-C event *as the latest event* would re-veto structures that have already advanced — the broader evidence set is the faithful reading (pinned from the Stage-1 pilot; the viewer's deterministic screen implements exactly this set). The veto is **binary** (eligible / not), but the sizing-band ceiling it hands downstream differs by regime: a confirmed trend (`markup`/`markdown`) and a post-phase-C continuation branch (`reaccumulation`/`redistribution`) authorize the **upper** band, the post-phase-C base regime (`accumulation`/`distribution`) the **conditional-top** band, pre-phase-C the **conditional floor** — RISK enforces these. The Wyckoff veto does not consume MCP-delivered fields directly; it consumes the operator-confirmed regime/phase reading produced by WYCKOFF's propose-confirm protocol from MCP price/volume metrics. The trigger's data-honesty floor is that the runtime never asserts a regime/phase reading as confirmed unless the operator has explicitly confirmed it in the current session — propose-confirm is a per-session protocol, not a memory-persisted state.

**Dealer-timing veto — entry refused when the dealer regime is too unstable to support new exposure at this moment.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Per-ticker DGPI tier and flip-zone classification from DEALER (the ticker layer, not macro); macro hostile-macro flag from DEALER (the macro layer); per-ticker near-flip flag from DEALER; per-ticker dealer `confidence` (high / medium / low / invalid) from DEALER |
| Firing condition | The macro hostile-macro flag is active AND the candidate is **bullish** long-premium (long call, call debit spread) — a hostile macro is a falling/unstable tape, so it refuses the bullish direction while the directionally-aligned bearish structures (long puts, put debit spreads) remain eligible as the redirect; OR the per-ticker dealer regime is **adverse to the candidate's direction** — for a bullish candidate, DGPI reads hostile (≤ -30, ≤ -`DGPI_STRONG_BAND` per DEALER's Appendix) AND spot is well below its own flip; for a bearish candidate, the mirror (DGPI strongly supportive ≥ +30 AND spot well above its own flip — the per-ticker bearish-mirror DGPI band per DEALER's Appendix); OR the per-ticker dealer `confidence` reads `invalid` for a candidate that requires dealer regime confirmation (direction-neutral — absent data refuses either direction); OR the macro layer reads as stale dealer data without operator acknowledgment (direction-neutral) |
| Behavioral consequence | New long-premium entry refused. Eligible alternatives surfaced per GUARDRAILS' hostile-macro eligible-set redirect (CSPs, hedges, LEAPs, opposite-direction structures). The refusal is structural, not timing-defer — the candidate doesn't enter a "wait-and-retry" state, it's dropped from this screening run entirely |
| Enforcement venue | `KAPMAN_GUARDRAILS_v4.0.md` enforces the macro-level refusal (the hostile-macro composite is GUARDRAILS' canonical refusal mechanism); `PASS1_SCREENING_v4.0.md` enforces the ticker-level refusal at eligible-set determination; `REPORT_FORMAT_v4.0.md` renders the redirect |

The dealer-timing veto and the Wyckoff veto are independent — either can fire alone, and either firing is sufficient to refuse the entry. When both fire, the report surfaces both reasons in the rationale because they answer different questions: Wyckoff says "the structural setup is wrong," dealer-timing says "even if the setup were right, this isn't the moment." The macro layer's hostile-macro flag is the dominant case in practice — when SPY is below its flip with DGPI ≤ `HOSTILE_MACRO_DGPI_MAX` (currently -30), the macro-level refusal applies to every **bullish** long-premium candidate in the screening run regardless of how supportive any individual ticker's own dealer regime reads; the directionally-aligned bearish structures (long puts, put debit spreads) remain the eligible redirect, subject to the per-ticker stability checks. This direction-aware scoping reconciles the veto with GUARDRAILS' hostile-regime eligible-set, which lists long puts as "Eligible — directional alignment with regime." The near-flip flag does *not* fire the dealer-timing veto on its own; near-flip triggers a one-tier sizing step-down owned by RISK and required by GUARDRAILS, not an entry refusal. A near-flip candidate remains eligible regardless of DGPI tier — the firing condition above requires spot *well* beyond the flip on the adverse side, and near-flip is by definition neither side. When near-flip combines with a weakening or hostile DGPI the consequence is sizing, not refusal: RISK's dealer-tier narrowing floors the band on top of the near-flip step-down, and the Pass-2 fresh flip read resolves whether spot has actually slipped to the adverse side. `invalid` dealer `confidence` on a candidate ticker fires the veto by default because the trigger cannot confirm regime support from absent data — the operator can override by providing an explicit dealer-regime assertion, but the runtime does not infer support from absent data. Stale macro data fires the veto until refreshed because acting on a stale macro read against the screening run's full eligible set is a portfolio-level risk that the trigger declines to take silently.

**Spread-mandate — naked long-premium refused when IV/HV reads elevated.**

| Contract part | Specification |
|---|---|
| Inputs consumed | IV/HV band (cheap / neutral / elevated) from VOLATILITY for the candidate ticker, computed per VOLATILITY's source-authority rules (the Polygon producer's ATM `atm_iv` ÷ HV20, delivered as `iv_hv_ratio`; Pass 2 re-confirms on a fresh fetch); volatility-status label (FULL / LIMITED / INVALID) from VOLATILITY; IV rank tier from VOLATILITY (extreme tier reinforces the mandate even when IV/HV reads neutral) |
| Firing condition | IV/HV band reads elevated (≥ 1.20 per VOLATILITY's Appendix); OR IV/HV band reads neutral AND IV rank tier reads extreme (75-100 band per VOLATILITY's Appendix); OR volatility-status label reads INVALID for a candidate selecting between naked long-premium and spread structures; OR IV/HV band reads *Needs chain validation* because the Pass 2 re-confirm is unavailable |
| Behavioral consequence | Naked long-call and naked long-put structures refused for new directional entries. A vertical spread structure (call debit spread for bullish direction, put debit spread for bearish direction) is mandated. The candidate remains in the eligible set; only the structure is constrained. Sizing denominator shifts from underlying-notional (used for naked) to spread-risk (debit paid), per RISK's sizing logic |
| Enforcement venue | `PASS2_VALIDATION_v4.0.md` enforces the structure constraint during validation; `RISK_v4.0.md` consumes the resulting structure choice to determine the sizing denominator; `REPORT_FORMAT_v4.0.md` renders the spread structure and surfaces the *"Spread mandated — elevated IV/HV"* annotation |

This is the trigger formerly known as `iv_forbids_long_premium` in the v2.3 source (SIGNAL_008's third sub-condition). The v3.0 rename to *spread-mandate* reflects the trigger's actual behavioral consequence — it doesn't forbid long-premium directional exposure, it mandates that long-premium be structured as a defined-risk spread rather than as a naked long. The distinction matters because a naked long call and a call debit spread express the same directional view; the spread-mandate is about *how that view gets expressed* under expensive-IV conditions, not about whether the view is allowed. The trigger inherits VOLATILITY's source-authority discipline in full — the elevated-band call is made from the Polygon producer's ATM `iv_hv_ratio`, and a Pass 2 re-confirm that cannot re-fetch a clean ATM IV/HV degrades the trigger to fire-by-default (spread mandated) rather than fail-open (naked allowed). The extreme IV rank tier reinforcement clause exists because IV rank measures regime against the ticker's own history while IV/HV measures forward IV against realized — a ticker can have a near-1.0 IV/HV ratio (realized has caught up to implied) while still sitting in an extreme historical IV regime where any premium-paying structure carries elevated mean-reversion risk; in that case, the spread mandate fires off IV rank alone. Operator override is available per GUARDRAILS but is rare in practice — the spread-mandate is one of the most evidence-grounded triggers in the system because it's anchored to a quantitative IV/HV band rather than a regime judgment, and the runtime defaults to honoring it. When override is invoked, the report surfaces the override prominently in the rationale because spread-vs-naked is a sizing-band-defining choice that materially affects the position's risk profile.

**Stop alert — hard-trigger broker-actionable alert when the underlying reaches a thesis-breakdown level.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Operator-confirmed Wyckoff structural levels for the position's ticker **on the position's unfavorable side** — for a long, the downside support (support shelf below an accumulation range, range floor, last point of support `lps`); for a long put, the upside supply (resistance shelf, last point of supply `lpsy`); per-ticker dealer wall levels from DEALER (put wall below spot is a candidate floor for a long; call wall above spot for a long put); **the viewer's forward-tested unfavorable-side target + calibrated hit-rate (`pt_down_*` for a long / `pt_up_*` for a long put, + `*_prob`) from the §A1 Pass-1 handoff carried in position context, when present and fresh per `as_of`/`data_through` — confidence-annotation input only, not a price source**; current option chain snapshot from Schwab/Polygon MCP for the position's strike and expiration; current Greeks (delta, gamma) from the chain snapshot; current underlying spot |
| Firing condition | The alert is *defined* at the moment of position recommendation or post-entry review — not fired in the screening sense. The trigger's job is to compute and surface the alert level and the accompanying output fields. The "firing" event is the operator's broker subsequently tagging the underlying alert; SIGNAL's contract is to produce the level and the accompanying option-price estimate and trail-stop values for the operator to set in their broker |
| Behavioral consequence | The report surfaces four fields per position: (1) underlying alert price, anchored to a regime-confirmed structural level on the unfavorable side; (2) estimated option price at the underlying alert level, computed in-conversation by delta-gamma approximation with explicit accuracy band; (3) trail-stop value on a bid/ask basis (Fidelity-compatible); (4) trail-stop value on a mark basis (Schwab-compatible). The operator sets the alert and trail-stop in their broker; SIGNAL does not place orders. When a fresh viewer forward-tested target on the unfavorable side is present in position context, its calibrated hit-rate is carried as a confidence annotation on field (1) — *"…— viewer forward-tested hit-rate ~Z%, as-of [date]"* — when the forward-tested target confluences with the structural anchor (a near-coincidence tolerance — the forward-tested target sits at or adjacent to the structural level; its exact width is `FORWARD_TEST_CONFLUENCE_BAND_PCT` per SYSTEM_PARAMS (provisional ±0.5% of spot)); when they diverge beyond it, both levels surface for context and the operator weighs the divergence. The annotation is context; the structural+validated level remains the broker order, per the anti-hallucination floor |
| Enforcement venue | `PORTFOLIO_MGMT_v4.0.md` is the consumer for live-position monitoring; `REPORT_FORMAT_v4.0.md` renders the four-field output in the position's exit-plan row |

The Stop alert is a *broker-actionable* trigger, meaning its output is meant to be set in the operator's broker as an actual alert and trail-stop, not surfaced in the report as a runtime warning. This distinguishes it from the Regime exit advisory below, which surfaces in the report as a flag for operator review and is not actionable at the broker. The underlying alert level is anchored to a regime-confirmed level — typically a Wyckoff structural level (the support shelf below an accumulation range, a confirmed last point of support (`lps`) on the support side — or last point of supply, `lpsy`, on the supply side for a short-thesis position — the lower boundary of a trading range) or a DEALER-delivered wall level (a put wall just below current spot is a candidate floor where dealer hedging would be expected to provide near-term support; a break below it removes that support). The runtime does not derive the structural level itself — that comes from WYCKOFF (operator-confirmed) or DEALER (MCP-delivered). SIGNAL's contribution is composing the four output fields from those inputs. The estimated option price at the alert level uses a delta-gamma Taylor approximation around current spot — option_price_at_alert ≈ current_option_price + (delta × ΔUnderlying) + (½ × gamma × ΔUnderlying²) — and is labeled with an explicit accuracy band in the report: *"Est. option price at alert: ~$X.XX (Greeks-based projection; ±5-10% for near-term moves within ~1σ; less reliable for distant alerts or alerts >30 days out)."* The approximation degrades materially for far-from-spot alerts and for time-distant alerts because it ignores vol and time decay; the runtime acknowledges this rather than presenting the projection as precise. When current Greeks are not available from the chain snapshot (LIMITED chain quality, INVALID chain data, or refused chain validation), the option-price-at-alert field surfaces as *"Pending — option price projection unavailable"* per the GUARDRAILS data-quality vocabulary, and the underlying alert + trail-stop fields surface alone. Trail-stop values are computed as runtime arithmetic from the current bid/ask and current mark — the trail percentage or dollar amount is operator-set per position (defaults to the band in this file's Appendix), and the runtime produces both the bid/ask-basis trail (Fidelity-compatible) and the mark-basis trail (Schwab-compatible) so the operator can set the trail in whichever broker holds the position. The viewer's forward-tested target is never the alert level itself — it is a stale-prone snapshot projection; the alert level is the structural+validated anchor, and the viewer's calibrated hit-rate (when fresh) rides only as the confidence annotation on it.

**Profit target alert — hard-trigger broker-actionable alert when the underlying reaches a thesis-maturation level.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Operator-confirmed Wyckoff structural levels for the position's ticker **on the position's favorable side** — for a long, the upside supply/resistance (range high above an accumulation range, projected markup target, last point of supply `lpsy`); for a long put, the downside support (the distribution range-height objective / Ice-level projection — the downside analog of the projected markup target — and last point of support `lps`); per-ticker dealer wall levels from DEALER (call wall above spot is a candidate ceiling for a long; put wall below spot for a long put); **the viewer's forward-tested favorable-side target + calibrated hit-rate (`pt_up_*` for a long / `pt_down_*` for a long put, + `*_prob`) from the §A1 Pass-1 handoff carried in position context, when present and fresh — confidence-annotation input only, not a price source**; current option chain snapshot; current Greeks; current underlying spot |
| Firing condition | Same definition as Stop alert — the alert is *defined* at the moment of position recommendation or post-entry review. The trigger's job is to compute and surface the level and the accompanying outputs. The "firing" event is the operator's broker subsequently tagging the underlying alert |
| Behavioral consequence | The report surfaces the same four fields as Stop alert: (1) underlying alert price, anchored to a regime-confirmed structural level on the favorable side; (2) estimated option price at the underlying alert level, computed by the same delta-gamma approximation with the same accuracy band; (3) trail-stop value on a bid/ask basis; (4) trail-stop value on a mark basis. The operator sets the alert in their broker; SIGNAL does not place orders. When a fresh viewer forward-tested target on the favorable side is present in position context, its calibrated hit-rate is carried as a confidence annotation on field (1) — *"…— viewer forward-tested hit-rate ~Z%, as-of [date]"* — when the forward-tested target confluences with the structural anchor (a near-coincidence tolerance; exact width `FORWARD_TEST_CONFLUENCE_BAND_PCT` per SYSTEM_PARAMS); when they diverge beyond it, both levels surface for context and the operator weighs the divergence. The annotation is context; the structural+validated level remains the broker order, per the anti-hallucination floor |
| Enforcement venue | `PORTFOLIO_MGMT_v4.0.md`; `REPORT_FORMAT_v4.0.md` |

The Profit target alert mirrors the Stop alert's mechanics in every structural way — same four output fields, same delta-gamma approximation, same trail-stop bid/ask + mark dual output, same broker-actionable nature, same data-quality fallback behavior when Greeks are unavailable. The two triggers differ only in which side of current spot the underlying alert level anchors to. The pairing is intentional: every position recommendation surfaces both a Stop alert and a Profit target alert as a matched pair, so the operator has a complete exit plan at the moment of position entry rather than having to come back later to set the favorable-side alert. The favorable-side level is anchored to a Wyckoff structural target (the upper boundary of a confirmed accumulation range, a projected markup price objective, the next significant structural resistance) or a DEALER-delivered call wall (a strong call wall above current spot is a candidate ceiling). As with the Stop alert, the runtime does not derive the structural level; it consumes it from the upstream regime files and composes the four output fields. One asymmetry worth noting: the delta-gamma approximation tends to be slightly more reliable for favorable-side alerts than unfavorable-side alerts at equal distance, because favorable moves in long-premium positions are accompanied by expanding delta (the position becomes more directional as it works), which the gamma term captures; unfavorable moves are accompanied by contracting delta (the position bleeds), which the gamma term also captures but with the offsetting fact that gamma itself is shrinking. The accuracy band — ±5-10% within ~1σ near-term — is reported identically for both alerts to avoid suggesting a precision the approximation doesn't deliver.

**Regime exit advisory — non-actionable flag when the regime that authorized the position has decayed materially.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Operator-confirmed Wyckoff **regime + phase (A–E)** at the time of original entry (carried in position context); current operator-confirmed Wyckoff **regime + phase** per the current session's propose-confirm protocol; entry-time dealer regime snapshot (DGPI tier, flip-zone, dealer `confidence`) per DEALER as carried in position context; current dealer regime per the current session's MCP read; entry-time volatility regime snapshot (IV/HV band, IV rank tier, volatility-status label) per VOLATILITY as carried in position context; current volatility regime per the current session's MCP read |
| Firing condition | The entry-time Wyckoff **regime** has moved unfavorably **for the position's direction** per WYCKOFF's succession **graph, read relative to the position's direction** (the canonical graph is long-framed; the bearish edges are its mirror) — toward the regime set that closes the position's eligible direction (for a long: e.g. `markup` → `distribution`, `reaccumulation` → `redistribution`, `accumulation` → `ranging_undefined`; for a long put, the mirror — e.g. `markdown` → `accumulation` on an `sc` reversal) — OR the **phase has regressed** within the position's regime (phase D → B or D → A, i.e., a confirmed test/entry window has broken back into cause-building or stopping action); OR the entry-time DGPI tier has degraded by two or more tiers in the unfavorable direction (e.g., strongly supportive → near-neutral, moderately supportive → weakening); OR the entry-time spot-vs-flip relationship has crossed (e.g., position entered with spot well above flip, current spot is now well below flip); OR the entry-time volatility regime has crossed an IV/HV band against the position structure (e.g., position entered as naked long-premium when IV/HV was cheap, current IV/HV reads elevated and would now mandate a spread) |
| Behavioral consequence | The report surfaces an advisory flag in the position row: *"Regime exit advisory — [specific decay reason] — review."* No alert level is computed; no broker action is mandated. The operator decides whether to close, hold, or restructure. The advisory is informational, not a hard trigger |
| Enforcement venue | `PORTFOLIO_MGMT_v4.0.md` is the consumer for live-position monitoring and renders the advisory in the portfolio view; `REPORT_FORMAT_v4.0.md` renders the advisory text |

The Regime exit advisory is the only exit trigger that is *not* broker-actionable, by design. The other two exit triggers (Stop alert, Profit target alert) produce broker-set price levels because they're anchored to specific price thresholds where a thesis is structurally invalidated or matured. The Regime exit advisory is anchored to regime-state change, which is a continuous and often-judgment-bearing observation rather than a clean price threshold — and which the operator may legitimately choose to ride through because the position's time horizon exceeds the regime's volatility (LEAPs are the canonical example, but a holding period of 60+ days through a regime change can be entirely consistent with the original thesis if the underlying structural read still holds). The advisory therefore surfaces as a flag for operator review, not as a force-close. The four firing conditions are independent — any one is sufficient to surface the advisory — and when multiple fire, the report surfaces each reason in the advisory text so the operator can weigh them. The Wyckoff succession check requires both the entry-time confirmed regime/phase and the current confirmed regime/phase, which means the advisory can only fire when WYCKOFF's propose-confirm protocol has been executed in both sessions; if the current session has no confirmed Wyckoff reading, the Wyckoff branch of the advisory is suppressed and the report surfaces *"Regime exit advisory — Wyckoff regime not confirmed this session; review."* "Unfavorable" is judged relative to the position's direction (the catch-all is "any confirmed regime move that closes the long-premium eligible set for the position's structure"); a move deeper into the position's eligible regime does not fire the advisory (e.g. for a long, `accumulation`/`reaccumulation` → `markup` on `sos`/`jac`; for a long put, `distribution`/`redistribution` → `markdown` on `sow`/`ice_break`). Phase regression is scoped to D → B/A (a material loss of the confirmed entry window); a D → C retest is **not** a regression. The DGPI tier degradation threshold (two or more tiers) is deliberately not one-tier because one-tier regime drift is normal within a holding period and would produce too many false advisories; two-tier degradation is the band that historically correlates with thesis-relevant regime change. The volatility band crossing condition fires specifically when the current regime would have refused the original entry structure — a position entered as naked long-premium that now sits in an elevated IV/HV band where the spread-mandate would fire is structurally exposed to vol mean-reversion in a way the entry didn't account for. The advisory does not propose a specific alternative action; it surfaces the decay reason and lets the operator decide. The runtime contract is that the advisory must be honest about which decay condition fired, not generic — *"Regime exit advisory — review"* without a reason is a guardrail violation. The advisory does not block subsequent screening of the same ticker; a position with an active Regime exit advisory remains a candidate for portfolio-level review at any time, and a new screening run for the same ticker proceeds normally with current-session regime inputs (the advisory does not contaminate fresh screening).

**NO_TRADE consistency — refusal outputs do not carry active option structures.**

When a screening run produces a NO_TRADE recommendation for a candidate (because Wyckoff veto fires, dealer-timing veto fires, hostile macro is active without override, or a fallback path on degraded inputs forces refusal), the recommendation output does not carry an active option structure name (long_call, long_put, CSP, vertical_spread, etc.). The structure field reads as NONE or its equivalent in the output schema. This is an output-coherence rule: a recommendation that refuses entry is logically inconsistent with also specifying a structure to enter, and the runtime does not produce that combination. The rule applies regardless of which trigger produced the NO_TRADE — Wyckoff-vetoed candidates, dealer-vetoed candidates, hostile-macro-refused candidates, and fallback NO_TRADE candidates all surface with structure NONE. When the report needs to surface alternatives to a NO_TRADE (per the eligible-set redirect language in GUARDRAILS and in the Wyckoff/dealer-timing veto heuristics above), the alternatives carry their own structure specifications independent of the primary refusal — the primary structure remains NONE, the alternatives each name their own structure. The enforcement venue for this rule is the report rendering layer (`REPORT_FORMAT_v4.0.md`), with `PASS1_SCREENING_v4.0.md` and `PASS2_VALIDATION_v4.0.md` responsible for ensuring the runtime never composes a NO_TRADE-with-structure pairing in the first place.

**Alternative-confidence ordering — alternatives always carry strictly lower confidence than the primary recommendation.**

When a primary recommendation surfaces with one or more alternatives (eligible-set redirects per a veto, fallback alternatives per a degraded-data path, structure substitutions per the spread-mandate), the confidence score attached to each alternative is strictly lower than the primary's confidence score. The rule is structural: alternatives represent fallback paths the runtime considered but did not select, and ranking them at or above the primary's confidence would invert the recommendation hierarchy and corrupt downstream consumers that read confidence as a primary-vs-alternative discriminator. When the runtime's reasoning would naturally produce a near-equal confidence between primary and alternative (e.g., a strongly-supportive Wyckoff regime but a marginally-elevated IV/HV that mandates a spread, where the spread alternative is nearly as compelling as the naked long-premium would have been if eligible), the runtime resolves the tie by reducing the alternative's confidence to a margin below primary — typically a confidence delta of at least the band specified in this file's Appendix. The rule does not require alternatives to be confidence-ranked among themselves in any particular order beyond "all below primary," though convention is to surface alternatives in descending confidence order in the report. The enforcement venue is the report rendering layer (`REPORT_FORMAT_v4.0.md`); upstream runbooks (`PASS1_SCREENING_v4.0.md` and `PASS2_VALIDATION_v4.0.md`) are responsible for assigning confidence values that honor the ordering before the report consumes them.

**Fallback policy on degraded inputs — the runtime produces structured NO_TRADE output rather than emitting nothing.**

When a screening run encounters degraded regime inputs that prevent the normal trigger evaluation from proceeding — a candidate ticker for which dealer `confidence` reads `invalid`, volatility-status reads INVALID, the Wyckoff regime is unconfirmed (`UNKNOWN`) and the operator has not been asked, chain quality is below the validation threshold, or multiple regime inputs are simultaneously stale — the runtime does not silently emit nothing. It produces a structured NO_TRADE recommendation with the degradation reason named in the rationale: *"NO_TRADE — [specific degraded input] — review or refresh."* The structured fallback is the runtime's commitment to never produce an empty or absent output that the operator might mistake for "this candidate is fine." A WAIT alternative typically accompanies the NO_TRADE — a state that says "the candidate is structurally screenable but the inputs needed to evaluate it confidently are not currently available; recheck when refreshed" — and surfaces with confidence below the primary NO_TRADE. When a degraded path includes a structural opening (e.g., the candidate is in the `markup` regime per a confirmed Wyckoff reading, but volatility-status is INVALID so the structure decision cannot resolve between naked and spread), the fallback may surface a structure-mandated alternative (e.g., the spread alternative becomes the named eligible structure pending a fresh volatility read), but only when the structural alignment is unambiguous; ambiguous structural alignment defaults to WAIT alone. This heuristic inherits v2.3 SIGNAL_009's fallback discipline; the specific confidence values used at the v2.3 level (base 75/60, deltas of -20/-30) are MCP-internal output-formatting parameters documented in `engineering_only/SIGNAL_MCP_REFERENCE_v4.0.md` (forthcoming) — the runtime contract is that fallback output is structured and reason-bearing, not that the specific numeric values are load-bearing at runtime.

**Anti-hallucination floor on exact strikes, expirations, and numeric targets.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Pass 2 chain validation status (per `PASS2_VALIDATION_v4.0.md`); MCP-delivered option chain snapshot when Pass 2 has validated the chain; GUARDRAILS data-quality vocabulary |
| Firing condition | The trigger is *always active* — it is a standing floor, not a conditional gate. It fires (i.e., refuses to emit invented specifics) any time a trigger output would require a specific strike price, specific expiration date, specific entry price, specific stop-loss price, specific profit target, specific risk-reward ratio, or any other numeric specification that has not been sourced from validated MCP option-chain data for the ticker in the current session |
| Behavioral consequence | The output substitutes the would-be specific value with a zone, a qualitative label, or a *Candidate zone only* / *Needs chain validation* / *Pending* annotation per the GUARDRAILS data-quality vocabulary. Strikes are expressed as zones (e.g., *Candidate call zone $55–60*); expirations are expressed as DTE bands (e.g., *Candidate expiration [SWING_DTE_BAND]* for swing trades, *Candidate expiration [CSP_DTE_BAND]* for CSPs — values per SYSTEM_PARAMS); entry prices, stop-loss prices, and profit targets surface as zones anchored to the underlying alert levels owned by the Stop alert and Profit target alert triggers; risk-reward ratios are omitted entirely until the underlying components are validated |
| Enforcement venue | Every downstream consumer enforces this floor — `PASS1_SCREENING_v4.0.md` for Pass-1-only outputs, `PASS2_VALIDATION_v4.0.md` for validated outputs, `REPORT_FORMAT_v4.0.md` for the rendering layer. The floor is not the property of any single venue; it travels with every output |

This is the runtime descendant of v2.3 SIGNAL_005 (which forced six pricing fields to null at persistence) and inherits the same lineage as GUARDRAILS' VALIDATION_001 anti-hallucination commitment. The v2.3 mechanism — hardcoding null in tuple positions at the persistence boundary — is a backend implementation detail documented in `engineering_only/SIGNAL_MCP_REFERENCE_v4.0.md` (forthcoming). The runtime principle the mechanism enforced is the load-bearing piece, and it survives intact in v3.0 as this heuristic. The floor applies uniformly across every trigger in this file: the Wyckoff veto's eligible-set redirect names eligible structures but does not invent specific strikes; the spread-mandate names the structure (vertical spread) but expresses the legs as zones until Pass 2 validates; the Stop alert and Profit target alert surface underlying alert levels (which come from regime files, not from chain data) but use the delta-gamma approximation for option-price-at-alert and label it explicitly as an estimate. The floor is not a refusal to be useful — it's a refusal to launder unvalidated values into outputs that look authoritative. Operator override is not available against this floor; GUARDRAILS' override discipline covers regime gates and structure mandates, not data-honesty floors. An operator who needs an exact strike requests Pass 2 validation, not an override of the anti-hallucination floor.

**Wyckoff-event-derived directional fallback — direction resolves to BULLISH on confirmed `sos`, BEARISH on confirmed `sow`, NEUTRAL otherwise.**

| Contract part | Specification |
|---|---|
| Inputs consumed | Operator-confirmed Wyckoff events for the candidate ticker (`sos` / `sow` as the primary fallback inputs; other Wyckoff events — `spring`, `utad`, `lpsy`, `lps`, etc. — handled by their respective trigger heuristics rather than here) |
| Firing condition | The runtime needs to assign a direction (BULLISH / BEARISH / NEUTRAL) to a candidate for which other directional signals are absent or in conflict. The trigger fires as a *fallback* — it is not the primary directional resolution path; it activates only when the standard directional signals from price action, dealer regime, and volatility regime have not produced a clear direction |
| Behavioral consequence | Direction = BULLISH when the operator has confirmed a Sign of Strength (`sos`) event in the current session for this ticker. Direction = BEARISH when the operator has confirmed a Sign of Weakness (`sow`) event. Direction = NEUTRAL when neither event has been confirmed. The directional resolution flows to downstream structure selection (BULLISH → long call or call debit spread or CSP-on-the-name; BEARISH → long put or put debit spread; NEUTRAL → no directional position) |
| Enforcement venue | `PASS1_SCREENING_v4.0.md` consumes the fallback direction during eligible-set determination; `PASS2_VALIDATION_v4.0.md` consumes it during structure validation; `REPORT_FORMAT_v4.0.md` surfaces the fallback path in the rationale (*"Direction inferred from confirmed Wyckoff event: [`sos` / `sow`]"*) so the operator can see that direction came from the event fallback rather than from primary signals |

This heuristic descends from v2.3 SIGNAL_010, which gave `sos`/`sow` event-derived direction as a deterministic fallback. v3.0 preserves the same logic but anchors it to operator-confirmed events rather than to events claimed as MCP-detected — consistent with WYCKOFF's propose-confirm protocol. `sos` and `sow` are the canonical bullish and bearish event types in the Wyckoff vocabulary, and their directional reading is structurally unambiguous: `sos` indicates an accumulation-family regime's test confirmed (phase C→D) or `markup` strengthening (bullish); `sow` indicates a distribution-family regime's test confirmed (phase C→D) or `markdown` strengthening (bearish). Other Wyckoff events — `spring` (bullish, confirms accumulation phase C), `utad` (bearish, confirms distribution **phase C** — the short-side analog of a spring), `lpsy`/`lps` (continuation / entry-timing signals) — also carry directional implications but are handled by their respective trigger contexts (`spring`/phase-C confirmation is consumed by the Wyckoff veto heuristic; `utad` reinforces the bearish read for the Regime exit advisory and the Wyckoff veto's bearish phase-C gate). The directional fallback exists because the runtime needs a defined directional state for every candidate, and the absence of clear primary directional signals would otherwise produce ambiguous output. The fallback's *NEUTRAL* default is conservative — when the operator has not confirmed an `sos` or `sow` event, the runtime does not assign a direction from indirect signals (e.g., from DGPI sign or from price trend), because direction assignment without explicit Wyckoff confirmation in a system that already requires operator confirmation for the broader Wyckoff regime reading would create a back-door for unconfirmed regime assertions to flow into the recommendation. The operator can override the NEUTRAL fallback per GUARDRAILS by explicitly stating direction in the screening request.

## Workflow integration

**Position in the document hierarchy.** SIGNAL is tier T1 — a principle file. It owns the named-trigger vocabulary that converts regime reads into runtime decisions. Where the other T1 files (DEALER, VOLATILITY, WYCKOFF) each own a regime — dealer regime, volatility regime, Wyckoff regime, phase, and event reading — SIGNAL owns the *composites* that combine those regimes into specific firing conditions. **Entry point for every trigger evaluation.** Before any SIGNAL trigger evaluates for a candidate, the following must hold in the working context:

1. The candidate has a confirmed Wyckoff regime (and phase A–E where a range is active) reading for the current session — either the operator has confirmed Claude's proposed regime/phase per WYCKOFF's propose-confirm protocol, the operator has declared a regime/phase directly, or the runtime has surfaced *"Wyckoff regime unconfirmed"* (`UNKNOWN`) and the dependent triggers are degraded accordingly. The runtime does not silently proceed on an inferred regime.
2. The candidate has fresh DEALER outputs in the current session — per-ticker DGPI tier, flip-zone, near-flip flag, dealer `confidence`, and SPY-derived macro flags for the eligible-set determination. Stale or absent dealer data degrades dependent triggers per DEALER's stale-data contract.
3. The candidate has fresh VOLATILITY outputs in the current session — IV/HV band, IV rank tier, volatility-status label. The IV/HV band specifically requires the Pass 2 re-confirm (a fresh Polygon fetch) for the spread-mandate trigger to fire definitively rather than fire-by-default.
4. Override status is established for the current request — either no override is in effect (default), or an explicit operator override per GUARDRAILS' override discipline has been declared, named, and surfaced in the report. Override is per-request, not session-persistent.
5. For exit triggers specifically, position context must be available — the position's structure, strikes, expiration, entry date, entry-time regime snapshots, and current chain snapshot. Without position context, exit triggers do not produce output.

If any of these conditions fail, the affected trigger evaluates to its conservative-read fallback and the report surfaces the gap explicitly. SIGNAL does not produce trigger output on incomplete context.

**Delta-gamma approximation formula and accuracy band.**

| Element | Specification |
|---|---|
| Formula | `option_price_at_alert ≈ current_option_price + (delta × ΔUnderlying) + (½ × gamma × ΔUnderlying²)` |
| ΔUnderlying | `underlying_alert_price − current_underlying_spot`; positive for favorable-side alert in long calls and unfavorable-side in long puts; negative for unfavorable-side in long calls and favorable-side in long puts |
| Inputs required | `current_option_price` (from MCP chain snapshot — bid/ask midpoint or mark); `delta` (from chain snapshot Greeks); `gamma` (from chain snapshot Greeks); `current_underlying_spot` (from MCP quote); `underlying_alert_price` (from Stop alert or Profit target alert level resolution) |
| Accuracy band — near-term, within ~1σ of current spot | ±5-10% of computed value |
| Accuracy band — distant alerts (>30 days to alert tag) or far-from-spot alerts (>~1σ) | Materially worse; runtime surfaces the broader band in rationale |
| Report rendering language | *"Est. option price at alert: ~$X.XX (Greeks-based projection; ±5-10% for near-term moves within ~1σ; less reliable for distant alerts or alerts >30 days out)"* |
| Fallback when formula cannot resolve | *"Pending — option price projection unavailable"* surfaces in the option-price field; underlying alert + trail-stop fields surface normally |

The formula is a second-order Taylor expansion. It ignores vega (vol changes during the holding period) and theta (time decay between now and alert), and it ignores third-order Greeks (charm, vanna, vomma). For near-term moves within ~1σ of current spot, the second-order approximation captures most of the option's price response and the ignored terms are second-order corrections to the second-order approximation; the ±5-10% band reflects this. For distant alerts or alerts where the underlying needs to move several standard deviations, the ignored terms dominate and the approximation should not be relied on for precision — the underlying alert + trail-stop fields remain actionable, but the option-price field becomes a rough magnitude estimate at best.

**Trail-stop reference band.** Default trail-stop percentage for new positions; operator may override per position:

| Position type | Bid/ask basis (Fidelity-compatible) | Mark basis (Schwab-compatible) | Notes |
|---|---|---|---|
| Long call, long put (naked, normal sizing) | 25-35% trail | 20-30% trail | Mark-basis tighter because mark is mid of bid/ask and moves less per tick |
| Vertical spread (debit) | 30-40% trail | 25-35% trail | Spread trails wider because each leg's bid/ask spread compounds in the position |

**Confidence-delta band for primary-to-alternative ordering.** Per heuristic 8, alternative recommendations carry strictly lower confidence than the primary. The minimum delta between primary and alternative is:

| Scenario | Minimum confidence delta (primary − alternative) |
|---|---|
| Eligible-set redirect under a veto (Wyckoff veto, dealer-timing veto, hostile macro) | At least 0.10 |
| Structure substitution under spread-mandate (naked refused, spread offered as alternative) | At least 0.05 — spread is structurally close to naked, smaller delta is appropriate |
| Fallback alternatives under degraded inputs (WAIT alternative under NO_TRADE) | At least 0.15 — WAIT carries materially less conviction than NO_TRADE under a known-degraded data condition |
| Multiple alternatives ranked among themselves | No specified delta between alternatives; only the primary-to-alternative gap is constrained; convention is to surface alternatives in descending confidence order |

The deltas above are runtime reference values for the v3.0 implementation. The v2.3 source specified strict inequality only (no minimum delta); v3.0 introduces the minimum-delta band to prevent confidence-score noise from collapsing the primary-vs-alternative distinction at the rendering layer.
